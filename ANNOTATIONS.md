### 2.2 `Config.py`

**亮点**：
- 超参数集中管理，所有可调项清晰标注了默认值和物理意义
- 对每个参数都写了类型注解（`: int`/`: float`/`: tuple`），IDE 友好

**批注**：

> `n_bins = (6, 6, 12, 8)` — 为什么角度维度 (12) 比位置 (6) 多一倍？因为 CartPole 中杆子角度变化对平衡更重要，更细的离散化能捕捉更细微的倾斜差异。这是一个经过思考的选择，而非随意设定。

> `epsilon_decay = 0.995` — 这是**指数衰减**。衰减 500 回合后 `ε = 1.0 × 0.995^500 ≈ 0.082`，仍在 `epsilon_min=0.01` 之上。ε 衰减路径是合理的：前 200 回合大量探索，后 300 回合逐步收敛到利用。

> `gamma = 0.99` — CartPole 本质上是一个「尽量延长持续时间」的任务（每步 reward=1）。γ 接近 1.0 意味着智能体高度重视远期回报，这与任务目标一致。

**观察**：
- `seed = 42`（道格拉斯·亚当斯的"生命、宇宙以及一切的终极答案"），经典的选择 😄
- `pg_lr` 和 `learning_rate` 都是 0.1 —— 这是一个不错的默认值，但**在实际调参中这两个通常需要不同的值**。策略梯度更新是回合级的（方差大），往往要更小的学习率。

**潜在改进**：
- 可以加入 `__repr__` 方法，方便打印当前配置
- 可以加 `from_dict()` / `to_dict()` 实现 JSON 序列化（方便保存实验配置）

### 2.3 `utils.py`

**亮点**：
- `_build_bins()` 的等距分箱实现干净利落
- `STATE_RANGES` 明确注释了每个维度的来源（CartPole 的物理限制）
- `discretize()` 包含了 `clip` 保护，防止极值越界

**批注**：

> `STATE_RANGES` 中的速度范围 `(-3.0, 3.0)` 是**经验值**（CartPole 速度无硬性上限）。好的智能体通常将速度控制在远小于 ±3.0 的范围内(实际可能 ±2.0 以内)。这意味着大部分箱子被浪费了。如果使用**自适应分箱**（如 tile coding），能更高效利用有限的分箱容量。

> `_build_bins()` 返回 `n-1` 个内部边界 → `np.digitize` 将连续值映射到 `n` 个箱子。标准做法，没有问题。

> `discretize()` 中 `d = min(d, len(bins[i]))` — 这个保护仅在值**恰好落在最后一个边界之上**时起作用（`digitize` 返回 `n` 而非 `n-1`）。在有 `clip` 的前提下，这个保护正常情况下不会被触发，但留着是好的防御性编程。

### 2.4 `QLearningAgent.py`

**算法正确性**：✅ 与 Sutton & Barto (2nd Ed.) 公式 6.6 一致

```python
# 公式：
Q(S,A) ← Q(S,A) + α [ R + γ·max_a Q(S',a) - Q(S,A) ]
```

**批注**：

> `q_table = np.zeros(cfg.n_bins + (2,))` — 形状 `(6, 6, 12, 8, 2)` = 6912 个状态-动作对。对于 CartPole 这个数量足够了，但要注意：实际使用中大部分状态永远不会被访问（例如位置接近 2.4 + 速度接近 3.0 的组合），真正有效的 Q 表条目远少于此。

> `choose_action()` 的 `training=False` 参数设计得很好 —— 训练时用 ε-贪婪，评估时纯贪婪。但当前项目中训练结束后智能体即被释放，所以评估模式实际上没有被使用。如果将来加入「阶段性评估」功能（比如每 50 回合跑一次无探索的测试），这个参数就派上用场了。

> `update()` 中 `td_target = reward + (0 if done else self.gamma * np.max(self.q_table[ns]))` —— 处理 done 的逻辑是正确的。终止状态没有合法的下一状态，Q(S',a) 应视为 0。这里用三元表达式 `0 if done else ...` 清晰地表达了这一点。

**潜在改进**：
- 可以加入**乐观初始化**（`q_table` 初始化为一个小正值如 +1），在早期鼓励探索。当前 Q 表全零 + ε=1.0 已经足够随机，但乐观初始化有时会加速收敛
- 可以使用 `np.unravel_index(np.argmax(q[s]), ...)` 替代 `np.argmax(q[s])` 来显式获取动作索引（目前只取最后一维索引，恰好是动作维，工作正常）

### 2.5 `SARSA.py`

**算法正确性**：✅ 与 Sutton & Barto (2nd Ed.) 公式 6.5 一致

```python
# 公式：
Q(S,A) ← Q(S,A) + α [ R + γ·Q(S',A') - Q(S,A) ]
```

**批注**：

> SARSA 的关键特征是 `update()` 签名比 Q-Learning **多一个 `next_action` 参数**。这个设计强迫调用者 (训练循环) 在更新前先选好 A'，从而保证更新使用的是「实际执行的策略」而非「最优策略」。这是 SARSA on-policy 性质的正确实现。

> `update()` 内部 `ns + (next_action,)` 的元组拼接用法与 Q-Learning 的索引一致，Q 表索引为 `(x, v, theta, omega, action)` 五元组。代码风格统一，好的工程习惯。

> SARSA 在随机环境中表现通常比 Q-Learning 更**保守/安全**，因为它在更新时考虑了自己探索策略的影响。在 CartPole（确定性环境）上，SARSA 和 Q-Learning 应该收敛到相似水平，但学习路径可能不同。这是值得实验对比的有趣现象。

### 2.6 `PolicyGradientAgent.py`

**算法正确性**：✅ REINFORCE（Sutton & Barto §13.3）

```python
# 核心更新：
θ ← θ + α · G_t · ∇_θ ln π(A_t|S_t)
```

**批注**：

> 这是整个项目中最复杂的文件。从**零手写**神经网络前向传播、softmax、反向传播、梯度计算，没有依赖 PyTorch/TensorFlow。每一个矩阵运算都是显式的（`np.outer`, `W2.T @ dlogits`），这种「裸露」的实现非常有教育意义——逐行拆解策略梯度的反向传播过程。

> `_forward()` 使用 **tanh** 作为隐藏层激活函数（而非更现代的 ReLU）。对于 CartPole 这个规模的任务 tanh 足够了，但 ReLU 通常训练更快。如果将来扩展到更复杂的环境（如 MountainCar），换 ReLU 会更好。

> `update()` 中的回报标准化 `(returns - mean) / (std + 1e-8)` 是一个**关键技巧**——不是教科书的必要部分，但实际上几乎每个 REINFORCE 实现都会这样做。标准化大幅减小梯度方差，加速收敛。这里加了 `1e-8` 防止除零，防御性编程到位。

> 当前 `use_baseline = True` 但**实际上基线并没有被使用**！在 `update()` 方法中：
> ```python
> returns = (returns - np.mean(returns)) / (np.std(returns) + 1e-8)
> ```
> 这行做的是回报**标准化**（以均值为中心、单位方差），而非**基线减法**。
> 
> 真正的基线应该是学习一个状态价值函数 V(s) 或使用滑动平均回报 `R̄`，然后用 `G_t - b(s_t)` 替代 `G_t`。当前 "use_baseline" 这个参数名是**轻微误导**——它实际上控制的是回报标准化，而非传统意义上的基线。这与 `小车杆prompt.txt` 中提到的 `α(R_t - R̄_t)` 原始需求有一定区别。

> `clear_policy()` 重新随机初始化网络参数。这个设计满足「训练结束后清空策略」的需求。值得注意的是：**清空 = 重新随机初始化 ≠ 销毁对象**，所以同一个 agent 可以再次被训练（虽然当前代码在训练后 agent 被 GC 回收了）。

**潜在改进**：
- 实现真正的 `use_baseline`：维护一个状态价值估计 `V(s)`（或简单使用滑动平均回报 `R̄`），在更新时使用 `G_t - R̄`
- 加入梯度裁剪 (gradient clipping) 防止数值不稳定
- 考虑使用交叉熵梯度简化计算：`dlogits = probs; dlogits[action] -= 1; dlogits *= -1` 等价于当前实现

### 2.7 `cartpole_rl.py`（主程序）

**亮点**：
- 交互式终端 UI 实现精良——菜单、参数修改、训练-绘图一气呵成
- 三种算法的训练循环三个函数，结构清晰平行
- CLI 模式支持 `python cartpole_rl.py q_learning 500` 快捷启动
- `plot_results()` 的图表设计用心：原始散点 + 滑动平均 + 均值线 + 参数标注框

**批注**：

> `train_policy_gradient()` 中每个回合都用 `make_env()` 创建新环境然后 `env.close()`。这比复用同一个环境更安全（避免状态泄漏），但有一定开销。对于 CartPole 这个开销可以忽略不计。

> `train_sarsa()` 的**「预选第一步动作 → 循环内选下一步动作」**模式是 SARSA 的标准训练循环结构（见 Sutton & Barto §6.4 的伪代码）。实现正确。特别注意到在 done=True 时，`next_action` 虽然被选中但实际不会被使用（因为循环立即退出），没有浪费/错误。

> `plot_results()` 中 `window = min(20, len(rewards) // 5 or 1)` —— 当 rewards 少于 5 个时，`len // 5 = 0`，`or 1` 防止 window=0 导致除零。边界条件处理细腻。

> 交互式参数修改部分的 `cmd` 编号系统有一个**编号重载**的小问题：`cmd == '5'` 在 PG 模式下是 `use_baseline`，在 SARSA/Q-Learning 模式下是 `epsilon`。同号不同义，虽然菜单显示时会区分，但理解上需要一点适应期。

> 训练完后**没有保存 Q 表或网络参数**的功能。如果希望在不同次运行之间保持策略，需要添加 `save_model()` / `load_model()` 方法。不过这与「清空策略」的需求目标有所冲突，需要设计上的权衡。
